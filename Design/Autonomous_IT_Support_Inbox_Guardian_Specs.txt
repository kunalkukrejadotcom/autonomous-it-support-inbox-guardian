1. Document Information

Purpose: This single document combines Functional Specifications (what the product does and why) and Technical Specifications (how it is built). It is written at a level of detail that allows any AI coding assistant (or developer) to implement the entire end-to-end solution without additional clarification.

Scope: Autonomous agent for a shared IT Support team inbox (itsupport@yourcompany.com or equivalent).

Hackathon Constraints Met: Built with Gemini 3 + Google Cloud Agent Builder/ADK + Arize Phoenix MCP (one partner track). Fully open-source, deployable on Cloud Run, 3-minute live demo ready.

Version History: v1.0 – Initial spec based on chat history and real-time research (May 18, 2026).

2. Executive Summary

The Autonomous IT Support Inbox Guardian is a truly agentic system (not a chatbot) that owns a high-volume shared IT support mailbox. It runs 24/7 on scheduled triggers or Gmail webhooks, autonomously triages, resolves routine tickets, escalates complex ones, logs actions, and visibly self-improves its accuracy using Arize Phoenix MCP. Key Differentiators

Zero user initiation per cycle — fully autonomous.

Delivers immediate business impact (SLA reduction, team productivity).

Live self-improvement visible in demo via Phoenix traces + LLM-as-Judge.

Feasible MVP in 5–7 days using Arize gemini-hackathon starter.

Expected ROI for any IT team: 70–80% reduction in manual triage time.

3. Functional Specifications

3.1 Product Overview

A production-grade autonomous agent that acts as the “first line of defense” for an IT support team inbox. It processes every new email in real time, executes real actions (replies, archives, ticket creation), and continuously refines its own decision logic.

3.2 Target Users and Business Value

Primary Users: IT support teams, helpdesk managers, MSP consultants.

Secondary Users: Any team using a shared Google Workspace inbox (HR, Facilities, Finance).

Value Metrics:

Average response time: <5 minutes for routine tickets.

Manual effort reduction: 70–80%.

Self-improvement: Resolution accuracy improves week-over-week (visible in dashboard).

3.3 Key Features and Capabilities

Autonomous email triage & resolution.

Auto-reply with KB-linked solutions.

Ticket creation + assignment in Google Sheets / Firestore.

Internal notifications (email/Slack).

Live dashboard with metrics.

Runtime self-improvement via own traces.

Audit log of every action.

3.4 Autonomous Operation Flow (Detailed Step-by-Step)

Trigger → Cloud Scheduler (every 5–15 min) or Gmail push webhook.

Read → Fetch all unread emails in shared inbox (Gmail API, label-based or query).

Classify & Prioritize → Gemini reasons over subject/body + historical KB.

Decide Action (autonomous branching):

Routine (password reset, printer issue) → Generate + send reply + archive.

Complex → Create ticket record + assign owner + notify team.

Execute → Perform Gmail send/archive, Sheets write, etc.

Log & Trace → Record outcome + full OpenInference trace.

Self-Evaluate → Query Phoenix MCP → Run LLM-as-Judge → Update internal rules.

Dashboard Update → Refresh metrics (emails processed, resolution rate).

3.5 Self-Improvement Mechanism

After each cycle the agent calls Phoenix MCP to retrieve recent traces.

LLM-as-Judge scores replies on helpfulness, completeness, tone.

Agent updates classification rules / reply templates in persistent memory.

Improvement visible in demo (before/after reply examples + accuracy graph).

3.6 User Interactions and Dashboard

Public Cloud Run URL with read-only dashboard (no login for demo).

Sections: Last run timestamp, Today’s stats, Live action log, Embedded Phoenix dashboard iframe, “Trigger now” button (for demo only).

End-users (IT team) can optionally mark replies “helpful / not helpful” via reply link to feed self-improvement.

3.7 Non-Functional Requirements

Latency: Full cycle <60 seconds.

Scalability: Handle 200+ emails/day.

Reliability: Retries on transient API failures.

Auditability: Every action logged with trace ID.

4.8 Authentication, Authorization, and Security Model

Service Account with Domain-Wide Delegation (scopes: https://mail.google.com/, Sheets, etc.).

Impersonate the shared IT support mailbox (must be a licensed user account or properly delegated Google Group).

OAuth2 service account credentials JSON stored as secret in Cloud Run.

Least-privilege: Only required Gmail/Sheets scopes.

Deploy ADK app to Cloud Run (one command from starter).

Trigger: Cloud Scheduler cron job calling /process-inbox endpoint.

Environment variables: Phoenix API key, service account JSON (secret), inbox email.

Cloud Run logs + metrics.

@tool

def read_unread_emails(inbox_email: str, max_results: int = 20) -> str:

    service = get_gmail_service()

    results = service.users().messages().list(userId=inbox_email, q="is:unread", maxResults=max_results).execute()

        email = service.users().messages().get(userId=inbox_email, id=msg['id']).execute()

            "id": msg['id'],

agent/main.py (Autonomous endpoint + CLI)

python

from fastapi import FastAPI

from google.adk.runners import Runner

from .root_agent import inbox_guardian

from .instrumentation import instrument

import os

app = FastAPI()

instrument()  # Phoenix tracing

runner = Runner(agent=inbox_guardian)

@app.post("/process-inbox")

async def process_inbox():

    result = await runner.run(

        input={"task": "Process all unread emails in the IT support inbox autonomously"}

    )

    return {"status": "completed", "actions_taken": result}

# For local CLI testing

if __name__ == "__main__":

    import asyncio

    asyncio.run(process_inbox())
